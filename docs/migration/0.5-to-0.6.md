# Migration — 0.5 → 0.6

This is the Wave 1 delta: everything that shipped between 0.5.0 and 0.6.0.
Follow the order listed here when rolling out to production — each step is
either a pure additive change (safe to skip initially) or has an explicit
back-compat window.

## 1. Streamable HTTP spec completion (W1-01)

**Routing change**. `GET /mcp` now serves the SSE notification stream per
MCP Streamable HTTP 2025-03-26. The legacy `GET /mcp/events` path is
retained as a back-compat alias through 0.6 and **will be removed in 0.7**.

Action items:
- Clients on the pre-2025-03-26 spec: no change required for 0.6. Migrate
  to `GET /mcp` before upgrading to 0.7.
- Clients on the 2025-03-26 spec: should already send SSE requests to
  `GET /mcp` and will work out of the box.

**Resumability**. Clients may now send `Last-Event-ID: <n>` when
reconnecting; the server replays backlog events with `id > n`. Each SSE
event carries an `id:` line.

**Header validation**. Non-initialize requests that carry a
`Mcp-Protocol-Version` header whose value does not match the session's
negotiated version are rejected with HTTP 400 + JSON-RPC `-32600`. Absent
header is still accepted for pre-2025-03-26 clients. Rejections increment
`clockify_mcp_protocol_errors_total{code="protocol_version_mismatch"}`.

## 2. Resources and Prompts capabilities (W1-04 + W1-05)

`initialize.result.capabilities` now includes two additional blocks when
the server is wired with a `ResourceProvider` (which is the default for
`cmd/clockify-mcp`):

```json
{
  "resources": {"subscribe": true, "listChanged": true},
  "prompts": {"listChanged": true}
}
```

Clients that special-case `tools` only should ignore unknown capability
blocks — this is spec-compliant forward-compat.

New method handlers:

- `resources/list`, `resources/read`, `resources/templates/list`,
  `resources/subscribe`, `resources/unsubscribe`
- `prompts/list`, `prompts/get`

Five built-in prompt templates: `log-week-from-calendar`, `weekly-review`,
`find-unbilled-hours`, `find-duplicate-entries`, `generate-timesheet-report`.
See [`docs/prompts.md`](../prompts.md) (if present) or the
`builtinPrompts` function in `internal/mcp/prompts.go` for the full list.

Five parametric resource URI templates: `clockify://workspace/{workspaceId}`,
`.../user/{userId}`, `.../project/{projectId}`, `.../entry/{entryId}`,
`.../report/weekly/{weekStart}`.

No client-side action required for clients that don't want these
features; existing tool-only clients continue to work.

## 3. Progress notifications + per-token rate limiting (W1-03 + W1-07)

**Progress notifications**. `tools/call` now respects an optional
`_meta.progressToken` parameter; long-running report tools
(`clockify_weekly_summary`, `clockify_detailed_report`, etc.) emit
`notifications/progress` once per fetched page when a token is supplied.
Clients that do not set `_meta.progressToken` see the exact same behaviour
as 0.5.

**Per-token rate limiting**. Two new env vars:

| Var | Default | Purpose |
|---|---|---|
| `CLOCKIFY_PER_TOKEN_RATE_LIMIT` | `60` | Calls per 60s window per authenticated `Principal.Subject`. `0` disables the layer. |
| `CLOCKIFY_PER_TOKEN_CONCURRENCY` | `5` | Max concurrent tool calls per subject. `0` disables. |

Both defaults sit comfortably under the global caps. Requests without an
authenticated principal (static-bearer or forward-auth without a subject)
fall back to the global layer only, so anonymous / single-tenant
deployments see no behavioural change.

**Breaking metric label change**. `clockify_mcp_rate_limit_rejections_total`
now carries an additional `scope` label (`global` | `per_token`). Dashboards
that group by the full label set must be updated. Backfill for existing
dashboards:

```promql
sum without (scope) (rate(clockify_mcp_rate_limit_rejections_total[5m]))
```

This expression reduces the old and new series to the same shape so
historical panels keep working.

## 4. OpenTelemetry tracing (W1-12)

Tracing is opt-in. Default `go build` is unchanged — the binary still
carries **zero** OpenTelemetry symbols (verified by a CI gate running
`go tool nm`). To enable tracing, rebuild with `-tags=otel` and set
`OTEL_EXPORTER_OTLP_ENDPOINT`:

```sh
go build -tags=otel ./cmd/clockify-mcp
OTEL_EXPORTER_OTLP_ENDPOINT=https://otlp.example.internal:4318 \
  CLOCKIFY_API_KEY=xxx \
  clockify-mcp
```

Spans are emitted at two sites: `mcp.tools/call` (with `tool.name` +
`outcome`) and `clockify.http` (with `upstream.endpoint`, `http.method`,
`http.status_code`). W3C `traceparent` propagation is enabled on the
outbound Clockify HTTP request so downstream services can stitch the
trace.

See [`docs/observability.md`](../observability.md) for the full tracing
section.

## 5. Schema tightening (W1-10)

Every Tier 1 + Tier 2 tool's input schema now declares
`additionalProperties: false`, bounded `page` / `page_size`, `date-time`
format on RFC3339 timestamp fields, and a pattern on hex color fields.
This is a **contract change only** — no runtime JSON-schema validator is
wired yet, so the server does not reject requests with extra fields. A
Wave 2 follow-up will add server-side enforcement.

Client action:
- Validating clients that honour `additionalProperties: false` may start
  rejecting their own requests if they were previously sending unknown
  keys. Strip the unknown keys.
- Non-validating clients see no change.

## 6. Operator manifests (W1-13 + W1-14)

Two new files under `deploy/k8s/`:

- `prometheus-rule.yaml` — Prometheus Operator `PrometheusRule` with the
  multi-window burn-rate alerts + `ClockifyMCPUpstreamUnavailable` +
  high-latency alert.
- `servicemonitor.yaml` — matching `ServiceMonitor` scraping `/metrics`
  on the existing `http` port.

Apply both with `kubectl apply -f deploy/k8s/` once your cluster has
Prometheus Operator installed.

## 7. Runbooks + troubleshooting

Four new runbooks (`high-latency`, `metrics-scrape-failure`,
`shutdown-drain-timeout`, `oom-or-goroutine-leak`), plus a new
[`docs/troubleshooting.md`](../troubleshooting.md) matrix covering the
most common client-side failure modes. No operational change required —
this is documentation only.

## Version bump

Increment the `version` const in `cmd/clockify-mcp/main.go` from
`0.5.0` to `0.6.0` when cutting the release tag. The version is embedded
in the `clockify_mcp_build_info` metric's `version` label, which the
release workflow's smoke test asserts on.
